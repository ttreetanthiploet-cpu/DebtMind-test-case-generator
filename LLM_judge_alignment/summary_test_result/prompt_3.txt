## Evaluation Criteria

### subject (หัวข้อทางการ)
- "good": Highly concise (≤ 12 words) and formal Thai bank memo title. States only the core request type (e.g., ขอปรับโครงสร้างหนี้, ขอรวมหนี้). Qualifiers should only be used if essential to distinguish a non-standard case (e.g., "สำหรับลูกค้าธุรกิจ").
- "acceptable": Formal and clear, but slightly wordy (13-18 words) or contains a common, non-essential qualifier (e.g., "สำหรับลูกค้ารายย่อย", "เพื่อลดค่างวด"). The core request is still clear.
- "invalid": Not formal, too long (>18 words), too vague, includes customer identifiers, or misstates the core action.

### objective (วัตถุประสงค์)
- "good": A direct, action-oriented statement (1-3 sentences) describing the specific task for staff. MUST clearly state the offer type (e.g., TDR) and all key parameters (e.g., accounts involved, new installment, new term). Avoids redundant intros like "วัตถุประสงค์คือ...". Any justification must be a brief, single subordinate clause (e.g., "เพื่อลดภาระการผ่อนชำระ").
- "acceptable": The core action for staff is clear, but the text is slightly flawed. It might start with a wordy phrase, have a justification that is too long/prominent, or miss a minor parameter (while still including the most critical ones).
- "invalid": Wrong action identified, missing multiple key parameters, or the justification completely obscures the core task.

### debt_cause (สาเหตุหนี้)
- "good": 2–4 sentences factually summarising the root cause. Follows source hierarchy: **1. Priority:** Accurately use reasons explicitly stated by the customer. **2. Fallback (Synthesis):** If no reason is stated, synthesise the cause by forming a clear, logical link between financial data and the problem. The synthesis must explicitly name the key factors (e.g., "high external debt," "days past due") to support its conclusion. It MUST NOT speculate on unstated personal events (e.g., "illness"). Citing data with placeholders (e.g., "{dpd}") is a strong indicator of quality but not strictly required if the reasoning is sound.
- "acceptable": Captures the main debt cause, but the logical link between data and the problem is weak or implied. For example, it lists data points (`sumOsNCB` is X, `dpd` is Y) without explaining how they connect to form the problem, or it correctly identifies the cause but fails to cite any supporting data.
- "invalid": Speculates on unstated personal events, contradicts the narrative, ignores customer-stated causes, or is uninformatively vague (e.g., "customer has financial problems").

### offer_suitability (เหตุผลสนับสนุน)
- "good": 2–4 sentences of sharp, evidence-based reasoning. It must directly connect the offer's key benefits to the core problem identified in the `debt_cause` field. The reasoning must be supported by citing the most critical data points (e.g., "The new installment of 396 THB is well below the customer's stated `maxPayment` of 800 THB, directly addressing their affordability limit and resolving the 25-day DPD `{dpd}`.").
- "acceptable": The reasoning is generally correct but less impactful. It might cite less critical data, make a generic claim, or the link back to the core `debt_cause` is implied rather than explicit.
- "invalid": The reasoning is flawed, cites irrelevant factors, makes unsupported claims, or contradicts the provided customer data.

### request_information (ข้อมูลเพิ่มเติม)
- "good": Accurately extracts all specific customer requests, conditions, preferred contact times, or follow-up items explicitly stated in the conversation; uses "ไม่มีข้อมูลเพิ่มเติม" only when the customer truly mentioned nothing additional.
- "acceptable": Captures the main request information but misses a minor detail; no fabrication; still useful for staff follow-up.
- "invalid": Misses major explicit customer requests, fabricates information not stated by the customer, or uses "ไม่มีข้อมูลเพิ่มเติม" when the customer did state specific requests.

### summary (สรุปภาพรวม)
- "good": A 3–5 sentence Thai paragraph that serves as a self-contained executive briefing. It MUST be a **freshly worded, cohesive narrative** that demonstrates true understanding by connecting the dots, not just listing them. It must fluidly synthesise: customer profile (who), core problem (why), proposed solution (what), and key justification (suitability). It must not read like a simple concatenation of other fields.
- "acceptable": Covers the required elements (who, why, what, suitability), but the synthesis is weak. It shows noticeable sentence or phrase-level reuse from other fields, or its structure feels more like a checklist than a flowing narrative.
- "invalid": Missing two or more key elements, is a direct copy-paste or simple concatenation of other fields, has an unprofessional tone, or is outside the 3-5 sentence range.
